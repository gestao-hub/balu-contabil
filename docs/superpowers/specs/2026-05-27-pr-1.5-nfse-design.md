# PR 1.5 — Configurações · NFS-e (design)

**Data:** 2026-05-27
**Status:** aprovado (brainstorming) — pronto para writing-plans
**Branch:** `feat/pr-1.5-nfse`
**Fontes:** PRD §8 (Configurações) + bloco `empresas_fiscais`/`municipios_nfse`; schema real `src/types/database.ts`. Depende da PR 1.4 (`EmpresaFiscalSchema` + `upsertEmpresaFiscalAction`).

## Contexto e decisão central

A aba **NFS-e** de `/configuracoes` configura a emissão de NFS-e da empresa. Decisão de design (aprovada): **o município da NFS-e é o do endereço da empresa** — não há seletor. Como `municipios_nfse` tem **1512 linhas e nenhum código IBGE**, o sistema casa `companies.municipio` + `uf` com `municipios_nfse` por **nome + estado**, normalizando acento/caixa. Por isso o endereço (cidade/estado) é obrigatório (PR de endereço obrigatório, ver PRD §6.7/§8).

Certificado digital A1 é a **PR 1.6** — aqui apenas exibimos aviso quando o município exige certificado.

## Escopo

Implementar a aba **NFS-e**: resolve o município pelo endereço, exibe a config derivada (read-only), e coleta os campos próprios da empresa (IM fiscal, série/numeração RPS, credenciais conforme o tipo de autenticação, toggle de ativação), persistindo em `empresas_fiscais` via o upsert da PR 1.4.

### Fora de escopo
- Upload/validação de **Certificado A1** → PR 1.6 (apenas aviso aqui).
- Credenciais "portal municipal" (`login_responsavel`/`senha_responsavel`/`token_portal`) — não usadas no v1.
- Flags de status de habilitação (`rps_liberado`, etc.) — adiadas (PR 1.4 spec).
- Emissão de nota em si (PR 2.1).

## Modelo de dados

`municipios_nfse` (1512 linhas; sem IBGE) — colunas relevantes: `municipio`, `estado`, `provedor`, `autenticacao`, `cancelamento`, `requer_liberacao_rps`, `requer_liberacao_webservice`, `requer_aidf`, `requer_cadastro_homologacao`, `emitir_nota_homol_antes_producao`, `credenciais_por_ambiente`, `requer_token_portal`, `im_zeros_esquerda`, `requer_cadastro_tomador`, `valor_iss_obrigatorio`, `cancelamento_so_portal`, `serie_rps_so_numeros`, `producao_disponivel`, `homologacao_disponivel`, `instrucoes_configuracao`.

`empresas_fiscais` — colunas tocadas por esta PR: `municipio_id` (FK), `inscricao_municipal`, `serie_rps`, `numero_rps_inicial`, `nfse_autenticacao_tipo`, `nfse_usuario_login`, `nfse_senha_login`, `nfse_token_api`, `nfse_habilitada`, `empresa_fiscal_ativada` (+ os de regime da PR 1.4).

### Valores de `autenticacao` (do município) → campos exigidos
A string pode conter combinações separadas por vírgula. Regra (substring case-insensitive):

| `autenticacao` contém | Campos exibidos (gravados) |
|---|---|
| `Login e Senha` | `nfse_usuario_login` + `nfse_senha_login` |
| `Token` | `nfse_token_api` |
| `Certificado digital` | (nenhum campo) — aviso "configure o Certificado A1 (aba Certificado — PR 1.6)" |
| `Não possui` / `null` | nenhum |

`nfse_autenticacao_tipo` recebe o valor de `autenticacao` do município.

### Flags derivadas
**Não são copiadas** para `empresas_fiscais`. Persistimos apenas `municipio_id`; as flags (`cancelamento_so_portal`, `requer_*`, urls de portal, `serie_rps_so_numeros`, etc.) são lidas via `municipio_id` → `municipios_nfse` quando necessário (fonte única, sem drift). Exibidas read-only na aba.

## Arquitetura

### 1. `src/lib/fiscal/municipio-nfse.ts` (novo) — helpers puros + resolução
- `normalizeNome(s)`: lowercase + remove acentos + trim (para comparar nomes).
- `credenciaisDaAutenticacao(autenticacao: string | null)`: retorna `{ login: boolean; token: boolean; certificado: boolean }` a partir da string.
- Testável isoladamente (vitest).

### 2. `src/lib/fiscal/municipio-nfse.server.ts` (novo, `server-only`) — `resolveMunicipioNfse`
- `resolveMunicipioNfse(supabase, municipio, uf)`: consulta `municipios_nfse?estado=eq.{uf}&deleted_at=is.null`, compara `normalizeNome(row.municipio) === normalizeNome(municipio)`, retorna a linha casada ou `null`.
- (Server-only porque usa o client Supabase; a lógica pura de normalização/credenciais vem do módulo do item 1.)

### 3. `src/types/zod.ts` — estender `EmpresaFiscalSchema`
Adicionar (todos `.nullable().optional()`, pois o upsert é parcial):
`municipio_id` (uuid string), `inscricao_municipal` (string), `serie_rps` (string), `numero_rps_inicial` (coerce number ≥ 0), `nfse_autenticacao_tipo` (string), `nfse_usuario_login` (string), `nfse_senha_login` (string), `nfse_token_api` (string), `nfse_habilitada` (boolean), `empresa_fiscal_ativada` (boolean).

### 4. `src/app/(auth)/configuracoes/actions.ts` — `upsertEmpresaFiscalAction`
Reaproveitado da PR 1.4 (upsert por `empresa_id`, escopo `owner_user_id`, cria no 1º save). Como o `EmpresaFiscalSchema.partial()` agora inclui os campos NFS-e, o mesmo action persiste tudo.

**Tweak necessário:** `normalizeRegimePatch` é tipado para `RegimePatch` (só campos de regime); trocar
`const data = normalizeRegimePatch(parsed.data)` por
`const data = { ...parsed.data, ...normalizeRegimePatch(parsed.data) }`
para que os campos NFS-e fiquem no objeto persistido **e no tipo** (a normalização de regime continua aplicada por cima).

### 5. `src/app/(auth)/configuracoes/NfseForm.tsx` (novo, client)
- Props: `initial` (registro `empresas_fiscais` parcial) + `municipio` resolvido (linha de `municipios_nfse` ou `null`) + `cidade`/`uf` da empresa.
- Se `municipio` é `null`: exibe aviso "NFS-e indisponível para {cidade}/{uf} (município não está na base)". Sem campos.
- Caso contrário: bloco read-only (provedor, autenticação, cancelamento, flags) + campos editáveis (IM, série RPS, número inicial RPS, credenciais conforme `credenciaisDaAutenticacao`, toggles). Mesmo **modo leitura/edição** (Editar/Salvar/Cancelar) das abas Dados/Regime.
- Submit → `upsertEmpresaFiscalAction({ municipio_id, inscricao_municipal, serie_rps, numero_rps_inicial, nfse_autenticacao_tipo, nfse_usuario_login, nfse_senha_login, nfse_token_api, nfse_habilitada, empresa_fiscal_ativada })` → toast.

### 6. `src/app/(auth)/configuracoes/page.tsx` — wire
- Já carrega `company` + `empresaFiscal`. Quando `tab==='nfse'`: chamar `resolveMunicipioNfse(supabase, company.municipio, company.uf)` e renderizar `<NfseForm initial={empresaFiscal} municipio={...} cidade={...} uf={...} key={company.id} />` (substitui o `TodoPanel` dessa aba).

## Fluxo de dados
`page.tsx` (server) → carrega empresa + resolve município NFS-e → `<NfseForm>` → submit → `upsertEmpresaFiscalAction` → valida (zod) → upsert `empresas_fiscais` → `revalidatePath('/configuracoes')` → toast.

## Tratamento de erro
- Sem empresa atual → a page já trata.
- Município não casado → aviso na aba (sem campos), não bloqueia o resto da config.
- Validação zod falha → toast.
- Erro Supabase → repassa `error.message`.

## Verificação
- `vitest`: `municipio-nfse.test.ts` (`normalizeNome` com acentos/caixa; `credenciaisDaAutenticacao` para cada valor e combinações) + schema (campos NFS-e aceitos no `EmpresaFiscalSchema.partial()`).
- `tsc --noEmit` zero erros.
- UI/manual: aba `nfse` da empresa de teste (Curitiba/PR — município casa, `autenticacao` null → sem credenciais); preencher IM + série RPS + ativar → salvar → persiste em `empresas_fiscais` (`municipio_id` setado). Município inexistente → aviso.

## Premissas documentadas
- Match por nome+UF normalizado (a base não tem IBGE). Homônimos resolvidos pelo UF.
- `numero_rps_inicial` via `z.coerce.number()` (input textual → número).
- A aba grava `municipio_id`; flags continuam em `municipios_nfse` (lidas por join).
- IM fiscal (`empresas_fiscais.inscricao_municipal`) é separada da IM em `companies` (aba Dados) — decisão aprovada, redundância aceita no v1.
