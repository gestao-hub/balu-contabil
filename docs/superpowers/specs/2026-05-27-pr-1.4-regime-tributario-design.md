# PR 1.4 — Configurações · Regime tributário (design)

**Data:** 2026-05-27
**Status:** aprovado (brainstorming) — pronto para writing-plans
**Fontes:** PRD §8 (Configurações) + bloco `empresas_fiscais` do modelo de dados; schema real `src/types/database.ts`.

## Contexto

A página `/configuracoes` tem 4 abas: **Dados da empresa** (✅ pronta), **Regime tributário**, **NFS-e**, **Certificado A1**. As três últimas são hoje um `TodoPanel` stub e **não estavam cobertas por nenhuma PR** do plano de 4 dias — apesar de serem dependência da emissão (PR 2.1) e do detalhe/cancelamento (PR 1.3), que leem `empresas_fiscais`.

Decisão: criar 3 PRs, uma por aba, ordenadas por dependência:
- **PR 1.4 — Regime tributário** (esta spec) — estabelece a base (`EmpresaFiscalSchema` + action de upsert + padrão de form) que 1.5/1.6 reusam.
- **PR 1.5 — NFS-e** (depende de 1.4) — caminho crítico p/ emissão.
- **PR 1.6 — Certificado A1** (depende de 1.4) — pipeline upload→Storage+n8n.

Princípio do projeto: **o banco é a fonte da verdade** (migrations 0001/0002 estão defasadas; ver `DB-DIVERGENCIA.md`). Esta spec alinha ao schema real.

## Escopo desta PR (1.4)

Implementar a aba **Regime tributário**: um form que cria/edita o registro `empresas_fiscais` da empresa atual, cobrindo regime, faixa de atividade (anexo), Fator R e CNAE.

### Fora de escopo (documentado)
- `inscricao_municipal` → vai para **PR 1.5** (agrupado com `serie_rps`/`numero_rps_inicial`, usado na NFS-e).
- Flags de status de habilitação (`rps_liberado`, `webservice_liberado`, `homologacao_liberada`, `producao_liberada`, `tomador_credenciado`, `aidf_solicitada`, `cadastro_homologacao_feito`, `nota_homol_emitida`) → **adiados** (geridos pelo workflow n8n, ainda inexistente).
- `regime_especial_tributacao` → adiado (campo avançado de NFS-e).
- Flags de município, credenciais NFS-e, certificado → PRs 1.5/1.6.

## Modelo de dados (schema real `empresas_fiscais`)

Colunas tocadas por esta PR: `empresa_id`, `owner_user_id`, `cnpj`, `regime_tributario`, `Code_regime_tributario`, `anexo_simples`, `usa_fator_r`, `cnae_principal`.

### Option set de regime (`Code_regime_tributario` + `regime_tributario`)

O regime é representado por DUAS colunas, gravadas em conjunto a partir de um único dropdown:

| Code (`Code_regime_tributario`) | Descrição (label do dropdown) | `regime_tributario` (Type_empresa) |
|---|---|---|
| `1` | Simples Nacional | `simples` |
| `2` | Simples Nacional — Excesso de sublimite de receita bruta | `simples` |
| `3` | Regime Normal (Lucro Real ou Presumido) | `simples` |
| `4` | Simples Nacional — MEI | `mei` |

> "MEI" = `Code_regime_tributario === '4'` (equivalentemente `regime_tributario === 'mei'`).

### Faixa de atividade econômica → `anexo_simples`

Input "Faixa de atividade econômica" (option set), persiste o **Anexo** na coluna real `anexo_simples`. Na carga, mapeia anexo→faixa para exibir o label amigável.

| Faixa (label) | `anexo_simples` |
|---|---|
| I Comércio | `Anexo I` |
| II Indústria | `Anexo II` |
| III Serviços comuns | `Anexo III` |
| IV Serviços com folha relevante | `Anexo IV` |
| V Serviços especializados | `Anexo V` |

## Arquitetura

### 1. `src/types/zod.ts` — `EmpresaFiscalSchema` (novo)
Schema Zod com os campos fiscais. Usado via `.partial()` no PATCH. Base reaproveitada por 1.5/1.6 (que estendem com NFS-e/certificado).

Campos desta PR:
- `regime_tributario: z.enum(['simples', 'mei'])`
- `Code_regime_tributario: z.enum(['1','2','3','4'])`
- `anexo_simples: z.enum(['Anexo I','Anexo II','Anexo III','Anexo IV','Anexo V']).optional()`
- `usa_fator_r: z.boolean().optional()`
- `cnae_principal: z.string().optional()`

Regra de consistência (refine): se `Code_regime_tributario === '4'` (MEI), então `anexo_simples` e `usa_fator_r` devem ser nulos/ausentes.

### 2. `src/app/(auth)/configuracoes/actions.ts` — `upsertEmpresaFiscalAction(patch)` (novo)
- Resolve `current_company` do profile do usuário autenticado.
- Busca `empresas_fiscais` por `empresa_id = current_company` (`maybeSingle`).
- Se **não existe**: `insert` com `empresa_id`, `owner_user_id = user.id`, `cnpj` (da `companies`) + `patch`.
- Se **existe**: `update ... .eq('empresa_id', current_company)` com `patch` + `updated_at`.
- Valida `patch` via `EmpresaFiscalSchema.partial()`.
- `revalidatePath('/configuracoes')`.
- Retorno: `{ ok: true } | { ok: false, error }`.

### 3. `src/app/(auth)/configuracoes/RegimeTributarioForm.tsx` (novo, client)
- Recebe `initial` (registro `empresas_fiscais` ou vazio).
- Estado controlado; segue o padrão de `DadosEmpresaForm`.
- Campos e regras de visibilidade:
  - **Regime** — dropdown com as 4 descrições; on change grava Code + `regime_tributario` derivado.
  - **Faixa de atividade econômica** — dropdown I–V; **visível só se Code ≠ 4**; persiste em `anexo_simples`.
  - **`usa_fator_r`** — checkbox; **visível só se `anexo_simples` ∈ {Anexo III, Anexo V}**.
  - **`cnae_principal`** — input texto (sempre).
- Submit → `upsertEmpresaFiscalAction(patch)` → toast (`useToast`) sucesso/erro.

### 4. `src/app/(auth)/configuracoes/page.tsx` (editar)
- Já carrega `company` + `empresaFiscal`. Renderizar `<RegimeTributarioForm initial={empresaFiscal}>` quando `active === 'regime'` (substituindo o `TodoPanel` apenas dessa aba). As abas `nfse`/`certificado` continuam stub até 1.5/1.6.

## Fluxo de dados
`page.tsx` (server) → carrega `empresas_fiscais` da empresa atual → `<RegimeTributarioForm>` → submit → `upsertEmpresaFiscalAction` → valida (zod) → insert/update → `revalidatePath('/configuracoes')` → toast.

## Tratamento de erro
- Sem empresa atual → mensagem "Nenhuma empresa selecionada." (a page já trata).
- Validação zod falha → `{ ok:false, error }` → toast vermelho.
- Erro do Supabase → repassa `error.message` no toast.

## Verificação
- `npx tsc --noEmit` zero erros.
- UI (Playwright/manual): abrir `/configuracoes?tab=regime`; (a) selecionar MEI esconde Faixa + Fator R; (b) selecionar Simples Nacional + Anexo III mostra Fator R; (c) Anexo I esconde Fator R; (d) salvar persiste em `empresas_fiscais` (cria registro no 1º save) e recarrega com os valores.

## Premissas documentadas
- O código `Code_regime_tributario` é o option set CRT do Bubble (1–4 acima), canonizado no PRD `empresas_fiscais`.
- `regime_tributario` guarda o `Type_empresa` em minúsculo (`simples`/`mei`), coerente com o dado real existente.
- Fator R só faz sentido para serviços (Anexo III/V) — não havia regra de visibilidade no Bubble, é uma melhoria aprovada.

## Enhancements pós-aprovação (mesma PR)

Ajustes acrescentados após o design original, validados em runtime:

- **Modo leitura/edição** no `RegimeTributarioForm`: a aba abre com os campos **bloqueados** e um botão **Editar**; ao editar, o rodapé vira **Salvar** + **Cancelar** (Cancelar reverte aos valores salvos e re-bloqueia; salvar com sucesso re-bloqueia). O **mesmo padrão foi replicado** ao `DadosEmpresaForm` (aba "Dados da empresa"), onde o **CNPJ permanece sempre bloqueado**.
- **Remontagem por empresa**: `page.tsx` passa `key={company.id}` aos forms para evitar estado stale ao trocar de empresa pelo seletor.
- **`deleted_at`**: leituras de `empresas_fiscais` filtram `deleted_at IS NULL` (convenção de soft-delete do projeto).
- **Endereço obrigatório + CNPJ** (mudança correlata, fora desta aba): `CompanySchema` exige `logradouro`/`municipio`/`uf` (CEP opcional) no cadastro e na edição; cadastro valida o CNPJ pelos dígitos verificadores via `CompanyCreateSchema` + `src/lib/validators/cnpj.ts`. Ver PRD §6.7 e §8.
